## 1. What We Are Building and Why

### The Business Problem

LittleJoys needs to simulate populations of real human decision-makers — specifically Indian parents navigating the purchase journey for children's nutrition products. The goal is decision intelligence: understanding which product configurations, pricing points, messaging strategies, and distribution channels will drive adoption, trial, and repeat purchase across different consumer segments.

This is not a research project. It is a commercial tool. The output must be:
- **Decision-useful:** Stakeholders must be able to act on simulation outputs
- **Distinct:** Different personas must produce meaningfully different behaviours
- **Consistent:** The same persona must behave coherently across multiple simulation turns
- **Explainable:** Decisions must be traceable to persona attributes and accumulated experience

### The Core Thesis

Existing persona systems — including our current implementation — are **static profile generators**. They produce detailed JSON objects that describe a person, but the person cannot perceive, react, remember, or change. This is like having a detailed character sheet but no actual character.

The proposed architecture builds **cognitive agents**: personas that perceive their environment, store experiences in a memory stream, synthesise those experiences into higher-level insights through reflection, and plan and act accordingly. The identity is deep and data-grounded (DeepPersona-style). The runtime behaviour is driven by accumulated experience (Generative Agents-style). The environment is grounded in the business problem (MiroFish-style).

## 2. The Three Source Papers

### 2.1 DeepPersona (NeurIPS 2025)

**Citation:** Wang et al., "DeepPersona: A Generative Engine for Scaling Deep Synthetic Personas," NeurIPS 2025 Workshop on LAW.

**Core contribution:** Two-stage taxonomy-guided persona generation.

**Stage 1 — Taxonomy Construction:**
- Mine 62,224 personalizable QA pairs from real human-ChatGPT conversations
- Classify each as: Personalizable / Partially Personalizable / Non-personal
- Recursively extract and hierarchically organise attribute phrases
- Merge semantically similar paths (>70% similarity threshold)
- Output: Human-Attribute Tree T with 8,496 unique nodes across 12 root categories

**Stage 2 — Progressive Attribute Sampling:**
- Anchor a stable core: age, location, career, personal values, life attitude, hobbies (sampled from bias-free tables, not LLM)
- Generate 3 life-story snippets (150-200 words each) from anchor
- Infer hobbies and interests from life stories (not stated directly)
- Progressive BFS traversal of T: at each step, randomly pick unexplored child node
- Three strata: near (most similar to anchors), middle, far — sampled at 5:3:2 ratio
- Each attribute value generated by LLM conditioned on P_<i (growing profile)
- Target depth: k > 100 attributes; optimal range 200-250
- Output: Narr(P) — 2000-character first-person narrative + structured JSON profile

**Validated results:**
- 32% higher attribute coverage vs PersonaHub
- 44% improvement in uniqueness
- 31.7% reduction in deviation from World Values Survey distributions
- 17% closer to Big Five personality distributions vs LLM-only citizens
- Optimal attribute depth: 200-250 (beyond 300, noise increases and performance drops)

**Key weakness:** Personas are static. No runtime behaviour, no memory, no environment.

### 2.2 Generative Agents: Interactive Simulacra of Human Behavior (UIST 2023)

**Citation:** Park et al., "Generative Agents: Interactive Simulacra of Human Behavior," UIST 2023.

**Core contribution:** Cognitive architecture for believable runtime agent behaviour.

**Three-component architecture:**

1. **Memory Stream:** Comprehensive record of all agent experiences. Each entry is a memory object with:
   - Natural language description
   - Creation timestamp
   - Most-recent-access timestamp
   - Importance score (LLM-rated 1-10 at creation)

2. **Retrieval:** At each time step, a retrieval function scores all memory objects on three dimensions:
   - **Recency:** Exponential decay over time since last access (decay factor 0.995/sandbox hour)
   - **Importance:** LLM-assigned score at creation (mundane=2, breaking up=8, college acceptance=10)
   - **Relevance:** Cosine similarity between memory embedding and current query embedding
   - **Final score:** `score = α·recency + β·importance + γ·relevance` (all weights = 1 in their implementation)
   - Top-K retrieved memories passed into LLM context window

3. **Reflection:** Periodic synthesis mechanism:
   - Triggered when Σ importance of recent observations > 150
   - Queries last 100 memory records
   - Asks LLM: "What are 3 most salient high-level questions to ask given these statements?"
   - For each question, retrieves relevant memories and generates an insight
   - Insight stored back to memory stream with pointers to cited observations
   - Agents reflect ~2-3 times per in-game day
   - Reflection trees: observations (leaf nodes) → inferences → higher-level abstractions (root)

**Planning:** Translates agent's summary description + recent experience into broad daily plan (5-8 chunks), then recursively decomposes into 5-15 minute action sequences. Plans stored in memory stream and retrieved alongside observations.

**Environment:** Smallville — hand-crafted Sims-like sandbox with tree data structure. Root = entire world, children = areas (house, cafe, park, store), leaves = objects (bed, desk, stove). Agents maintain individual subgraph of what they have perceived. Agent moves update environment state which is converted to natural language for LLM context.

**Validated results:**
- Full architecture TrueSkill: 29.89 (σ=0.72)
- No reflection: 26.88 — significant drop
- No reflection or planning: 24.5
- No memory at all: 21.21 — below human crowdworker baseline (22.95)
- Effect size vs no-memory: **8 standard deviations**
- Information diffusion: agent knowledge of key events grew from 4% to 32-52% without any user injection
- Hallucination rate on memory retrieval: 1.3% (6/453 responses)

**Key weakness:** Shallow 1-paragraph persona seeds. Environment is hand-crafted and domain-specific to Smallville.

### 2.3 MiroFish (2024)

**Source:** GitHub.com/666ghj/MiroFish — "A Simple and Universal Swarm Intelligence Engine"

**Core contribution:** End-to-end pipeline from seed documents to multi-agent simulation with knowledge graph memory.

**Key architectural patterns:**

**Ontology Generation (`ontology_generator.py`):**
- LLM analyses uploaded seed documents (PDF, markdown, text)
- Generates exactly 10 entity types (8 context-specific + 2 mandatory: Person, Organization)
- Each entity: name (PascalCase), description, 1-3 attributes
- Generates 6-10 relationship types (e.g., COMMENTS_ON, SUPPORTS) with source-target constraints
- Entities must be "real-world subjects capable of speaking on social media" — abstract concepts excluded

**Profile Generation (`oasis_profile_generator.py`):**
- Pulls entities from Zep knowledge graph
- Generates rich agent profiles: user_id, username, name, bio, persona (2000-char), age, gender, MBTI, country, profession, interested_topics
- Behavioral parameters per agent: activity_level (0-1), post/comment frequencies, active_hours, sentiment_bias, stance_positions, influence_weight

**Knowledge Graph (Zep Cloud):**
- Single source of truth for all persona attributes, episode memory, relationships, world state
- Seed memory: documents chunked (500-char chunks, 50-char overlap), batch-processed into graph
- Live memory: agent actions during simulation written back as episode text in batches of 5
- Retry logic with exponential backoff

**ReACT Report Agent:**
- Post-simulation analysis via 4 tools: insight_forge, panorama_search, quick_search, interview_agents
- insight_forge: decomposes questions → sub-questions → multi-dimensional graph retrieval
- interview_agents: live IPC calls to running agents for in-character responses

**Key strength:** Business-problem-conditioned world via ontology generation. Continuous memory write-back.
**Key weakness:** Personas are social-media archetypes (Twitter/Reddit behavioral types), not narrative-complete individuals.

### Tier 2 Generation Pipeline (tier2_generator.py) — What Actually Runs

**Step 1: Deterministic derivations** (no LLM, pure computation)
- `_derive_parent_traits()` — threshold-based enum assignment from psychographic values
- `_derive_budget_profile()` — formula: income × city_fraction × discretionary_ratio
- `_derive_decision_rights()` — priority-ordered rules from relationship attributes

**Step 2: LLM — Anchor Inference**
- Input: demographics + top 5 most-deviant psychographic attributes
- Output: `core_values` (list), `life_attitude` (2-3 sentences), `parent_motivations` (list)

**Step 3: LLM — Life Story Generation**
- Input: anchor payload + inferred values
- Output: 2-3 `LifeStorySnippet` objects with title, event, impact

**Step 4: LLM — Third-Person Narrative (300-500 words)**
- Input: all persona attributes + anchor + stories
- Validation: must reference at least 10 specific attribute values
- Uses Hindi-English code-mixing where culturally appropriate

**Step 5: LLM — First-Person Summary (120-150 words)**
- Input: city, age, children, employment, income, budget, dietary culture, shopping platform
- Diary-voice, starts with vivid morning scene from home city

**Step 6: LLM — Purchase Decision Bullets (6-8 bullets)**
- Input: decision_style, trust_anchor, value_orientation, budget figures, child attributes
- Max 20 words per bullet, directly actionable by product managers

## 6. Adversarial Critique — Where the Architecture is Likely to Fail

### FAILURE RISK 1 — Taxonomy Without a Constraint Graph (HIGH)

**The problem:** A taxonomy organises attributes hierarchically by semantic similarity. It says nothing about which attributes *cause* which other attributes, which combinations are *impossible*, or which values are mutually constraining.

Progressive BFS filling with 5:3:2 strata produces many attributes. But without a constraint graph, attributes are generated independently conditional only on narrative context. This means:

- `health_anxiety=0.9` + `supplement_necessity_belief=0.2` — psychologically incoherent. Your copula prevents this (ρ=0.55). Progressive fill does not.
- `budget_consciousness=0.85` + `organic_preference=0.9` + `income=4 LPA` — economically impossible. No constraint enforces this.
- Tier 3, joint-family, low-income persona with `indie_brand_openness=0.9` — contradicts documented Indian socioeconomic patterns.

**Consequence:** Deeper personas may behave *less* consistently than the current 145-field personas, because depth without dependency structure produces noise, not signal.

**What is needed:** A constraint graph or probabilistic graphical model layered on top of the taxonomy. The taxonomy tells you *what* attributes exist. The constraint graph tells you *which combinations are valid*.

---

### FAILURE RISK 2 — Progressive Fill Creates Textual Coherence, Not Behavioural Coherence (HIGH)

**The problem:** DeepPersona's progressive fill conditions each attribute on P_<i (growing profile). This produces *narrative consistency* — values that read as plausible together. But narrative consistency ≠ behavioural consistency.

A well-written persona can be "a budget-conscious mother who nevertheless spends heavily on organic products because only the best is good enough for her children." This reads as coherent. But under simulation, what wins: budget consciousness or organic preference? The narrative has no answer because it is post-hoc rationalisation, not a decision function.

**What is missing:** A causal chain: `backstory → values → preferences → decision rules`. Life events should *cause* value formation. Values should *constrain* preferences. Preferences should *determine* decisions.

The current system handles this with explicit priority ordering in `_derive_parent_traits`: "price" if `budget_consciousness > 0.65` takes priority over "nutrition" if `supplement_necessity_belief > 0.60`. These are decision rules. The proposed architecture has no equivalent for taxonomy-generated attributes.

---

### FAILURE RISK 3 — Reflection Will Hallucinate and Corrupt Identity (CRITICAL)

**The problem:** Reflection asks an LLM to synthesise 100 observations into 3 high-level insights. The LLM will:

1. **Over-generalise:** 10 observations of supplement ads produces "persona is deeply concerned about child nutrition" — conflating exposure with interest.
2. **Confabulate causality:** Unrelated observations become a coherent narrative. Two independent events become "persona is actively comparison-shopping."
3. **Corrupt Core Memory:** Once a false reflection exceeds the importance threshold (imp>8), it becomes immutable identity and influences all future behaviour. Compounding over a long simulation = increasingly fictional persona.

**Generative Agents achieved 1.3% hallucination rate** — but in a hand-crafted, tightly-scoped environment (Smallville, 25 agents, 2 game-days). Your system: 200 agents, auto-generated environment, ambiguous business stimuli.

**What is needed:**
- Confidence scores on reflections (separate from importance scores)
- Source citation validation — LLM-judge check that cited observations actually support the inference
- Bounded update rules — Core Memory attributes shift by ≤ ε per reflection cycle, not binary promotion
- Immutability partitioning — demographics and life history NEVER updated by reflection; only attitudes and preferences are mutable, and within bounds
- Forgetting/decay mechanism more principled than recency weighting

---

### FAILURE RISK 4 — Environment is Too Clean (HIGH)

**The problem:** Auto-generated environment ontology is a rational, complete, well-structured world. Real purchase environments are none of these:
- **Incomplete information:** Parents don't know what products exist or what they cost
- **Friction:** Out-of-stock, delivery delays, product tastes bad to the child
- **Stochastic events:** Random encounter at school gate, viral news article
- **Time pressure:** Decisions made under cognitive load, not in reflective state
- **Source credibility asymmetry:** Paediatrician ≠ mommy-blog (not currently modelled)
- **External shocks:** Product contamination news, competitor price cut, new entrant

A clean environment produces over-rational agents predicting ideal consumer behaviour, not real consumer behaviour. This is the same failure mode as classical economics.

**What is needed:** An explicit friction/noise layer: stochastic events, incomplete information states, cognitive load modifiers, source credibility weights.

---

### FAILURE RISK 5 — Cost and Latency Will Be Prohibitive at Scale (HIGH)

**Taxonomy construction:** ~$15-30 one-time per domain (acceptable).

**Per-persona generation:** 6-10 LLM calls per persona × 200 personas = 1200-2000 calls ≈ $12-20 (acceptable).

**Runtime loop:** Each step = perceive + retrieve + (conditional reflect) + plan + act = ~3 LLM calls minimum. 200 agents × 50 steps = 30,000 calls minimum. At $0.01/call = **$300 per simulation run**. For iterative stakeholder use: unsustainable.

**Latency:** Each step per agent = 5-10 seconds (sequential LLM calls). 200 agents × 50 steps = 50,000-100,000 seconds if sequential. Even with 50 concurrent agents = **17-33 minutes per run**. Stakeholders will not wait 30 minutes per scenario variant.

**Orchestration is the hardest problem.** Managing 200 concurrent agent state machines, each with memory streams, reflection triggers, environment perception, and inter-agent communication queues, is a distributed systems problem. Most teams underestimate by 5-10x.

## 8. Why This is Not Yet Predictive

**Critical distinction: plausible simulation ≠ calibrated prediction.**

**Plausible simulation:** "Given these attributes, here is a sequence of internally consistent, believable behaviours." This is what Generative Agents delivers (believability metric). This is what DeepPersona delivers (uniqueness and distribution fit metrics).

**Calibrated prediction:** "If you launch at this price to this demographic, 23% ± 4% will purchase within 6 months, and the primary barrier for segment A is price while for segment B it is trust."

**What is missing for prediction:**

1. **Ground-truth calibration data.** You need real-world outcome data (actual purchase rates, churn, WOM spread) to calibrate against. The architecture has no calibration step. There is no mechanism to verify that 200 generated personas produce the same aggregate behaviour as 200 real people.

2. **Outcome-model feedback loops.** After simulation predicts X% adoption, you need to observe actual adoption and feed the error back into persona or environment parameters. No feedback loop exists in the architecture.

3. **Behavioural base rates.** DeepPersona validates against attitude distributions (WVS, Big Five). Attitudes predict behaviour weakly — the attitude-behaviour gap is one of the most replicated findings in social psychology. You need validation against actual purchase behaviour data: trial-to-repeat conversion, category switching rates, WOM amplification rates.

4. **Sensitivity analysis.** A predictive system tells you: "The outcome is sensitive to assumptions X, Y, Z. If X changes by ±10%, prediction shifts by ±N%." No sensitivity analysis capability exists.

5. **Base-rate anchoring.** LLMs skew toward educated, English-speaking, digitally-literate populations. Without anchoring to behavioural base rates (e.g., "what fraction of Indian Tier 2 parents have ever purchased an online nutrition supplement?"), simulations will systematically over-estimate sophisticated behaviour.

**Honest stakeholder framing:** This system generates *decision-useful hypotheses* — "here are the likely barriers, segments, and sensitivity points." It cannot say "this will happen with X% probability" without a calibration pipeline.

## 9. Recommended v1 Build Scope

### The v1 thesis to prove

"This persona system produces more distinct, consistent, and decision-useful behaviour than naive LLM sampling."

This does not require the full 9-layer architecture. It requires proving three things:
1. Personas make different decisions at turn 15 than turn 3 (memory effect)
2. Personas respond differently to the same stimulus depending on prior experience (context sensitivity)
3. Stakeholders rate transcripts as more realistic and actionable than flat-prompt alternatives (business value)

### Include in v1

**Keep (do not modify):**
- Existing Tier 1 Gaussian copula — your strongest component, do not replace with taxonomy sampling
- Existing Tier 2 five-step LLM pipeline — produces good narratives, adequate for v1
- Existing schema — add memory fields, do not redesign

**Build (estimated effort):**

| Component | Effort | Priority | Notes |
|---|---|---|---|
| Implement `perceive()` | 3-5 days | P0 | Append stimulus to `episodic_memory` with LLM-scored importance |
| Implement `retrieve()` | 3-5 days | P0 | recency × importance × relevance scoring, return top-K |
| Implement `decide()` | 3-5 days | P0 | retrieve top-K + Core Memory → LLM decision with chain-of-thought |
| Basic reflection | 1 week | P1 | Trigger on Σ importance > θ, generate 3 insights, store with citations |
| 5 manual scenario stimuli | 3-5 days | P1 | Structured stimulus dicts for: discovery, price comparison, peer rec, GP visit, repeat purchase |
| Constraint checker | 1 week | P1 | 20-30 hard constraints on existing schema, run pre-simulation |
| Believability A/B test | 3-5 days | P1 | 5 personas × 2 conditions (v1 vs flat prompt) × 3-5 human raters |

**Do not build in v1:**
- Forum scraping and taxonomy construction
- Progressive attribute filling (copula is good enough)
- Environment auto-generation (manual scenarios are faster and more accurate)
- Agent-to-agent communication
- Core Memory promotion (too risky without understanding reflection quality)
- Knowledge graph backend (use in-memory Python dicts)
- Population-level WVS validation

### v1 Validation Test

Run 10 personas through a 20-turn simulation of the LittleJoys purchase journey:
`awareness → consideration → trial → repeat/churn`

Compare:
- **Condition A:** Current (static personas, no memory, no stimuli)
- **Condition B:** v1 (same personas + memory + reflection + 5 scenario stimuli)

Measure:
1. Do Condition B personas make different decisions at turn 15 vs turn 3? (memory effect)
2. Do Condition B personas respond differently to the same stimulus with different prior experience? (context sensitivity)
3. Do stakeholders rate Condition B transcripts as more realistic and actionable? (business value)

If yes to all three: validate and scale. If no: you have saved 6 months of building on a wrong assumption.

## 10. Validity Testing Framework

### 6-Axis Framework (full build)

**Test 1 — Intrinsic Quality** (no infra needed)
- Generate 10 domain personas
- LLM-as-judge: mean distinct attributes (target >50), uniqueness 1-5 (target >3.5), consistency check
- Compare against flat-prompt baseline

**Test 2 — Behavioural Distribution** (50+ personas)
- Run 6 World Values Survey questions across population
- KS statistic and Wasserstein distance vs real survey data
- Target: KS < 0.35 (DeepPersona achieved 0.30)

**Test 3 — Memory Coherence** (runtime loop needed)
- Run persona through 20 turns
- At turn 20, probe memory about turns 3, 7, 14
- Check retrieved memory matches stored observation
- Hallucination rate target: < 2%

**Test 4 — Reflection Validity** (reflection implemented)
- Run through interactions with escalating importance scores
- Trigger reflection
- Check: do reflections cite correct source observations?
- Check: are reflections higher-level inferences, not just summaries?

**Test 5 — Believability** (human evaluators)
- 5 personas × full vs. flat-prompt
- Human rankers, TrueSkill scoring
- Target: full architecture beats baseline with statistical significance

**Test 6 — Business Fit** (stakeholder review)
- LittleJoys domain, 3 persona transcripts, blind review
- Rate: realism, insight novelty, actionability
- Target: average > 4.0/5.0

## 11. Open Design Questions

These are unresolved and need decisions before building:

1. **Constraint graph design:** How do you represent attribute dependencies? Options: (a) hard rule constraints, (b) probabilistic graphical model, (c) causal DAG with intervention logic. Each has different expressiveness and implementation cost.

2. **Reflection confidence scoring:** What mechanism validates that a reflection actually follows from its cited observations? Options: (a) LLM-as-judge (expensive), (b) entailment model, (c) rule-based citation checker.

3. **Core Memory immutability boundary:** Exactly which attribute categories can be updated by reflection, and within what bounds? This needs a decision before the reflection engine is built — otherwise it will corrupt identity.

4. **Environment friction model:** How is incomplete information represented? How are stochastic events scheduled? What is the cognitive load model that degrades decision quality under stress?

5. **Cost ceiling per simulation run:** What is the acceptable cost in dollars per 200-agent, N-turn simulation? This determines how much of the runtime loop can be LLM-powered vs. rule-based.

6. **Calibration data source:** What real-world outcome data exists or can be collected to calibrate simulated purchase behaviour? Without this, the system remains hypothesis-generation, not prediction.

7. **Taxonomy merge strategy:** When augmenting the existing 145-field schema with forum-mined attributes, how do you avoid losing the copula's correlation structure? The taxonomy adds attributes; the copula defines their joint distribution. These need to coexist.
